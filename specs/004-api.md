# 004 — API

> Source: `openapi_specification.md` (OpenAPI 3.1, base `/api/v1`). That document is the contract; this spec governs how agents implement and depend on it.

## Purpose

Expose the domain and workflows as a versioned REST/JSON API that the web and field apps, reporting, and external accounting consume — with consistent auth, validation, pagination, filtering, sorting, idempotency, and error semantics.

## Mandated stack

- **Framework:** **NestJS** (modules, controllers, providers/services, guards, interceptors, pipes, exception filters, DI).
- **Validation:** **Zod** as the single source of truth for request/response schemas, integrated via **`nestjs-zod`** (`createZodDto`, global `ZodValidationPipe`). No `class-validator` DTOs.
- **API docs:** **`@nestjs/swagger`** emits the OpenAPI document and serves **Swagger UI**; `nestjs-zod`'s Swagger patch renders Zod schemas. Design contract remains `openapi_specification.md`.
- **Auth:** **Passport** (`@nestjs/passport`) with JWT — local/jwt/jwt-refresh strategies (see `005`), enforced by Nest `AuthGuard`s.
- **Runtime:** Node.js LTS + TypeScript.

## Requirements

- R1. Implement every endpoint in `openapi_specification.md` §4 across tags: Auth, Clients, Cylinders, Batteries, Movements, Sales, Accessories, SupplierLoans, Transfers, DeliveryNotes, Rates, RefillRates, Billing, Reports, Search, MasterData, Settings, Admin — organized as one **NestJS module per tag** (controller + service).
- R8. Define all request/response shapes as **Zod schemas** and derive Nest DTOs from them (`createZodDto`); a **global `ZodValidationPipe`** validates inputs to `422 VALIDATION_FAILED` with field-level `details[]`.
- R9. Serve the OpenAPI document + **Swagger UI** via `@nestjs/swagger` (e.g. `/api/docs`, JSON at `/api/docs-json`); the emitted spec is generated from the Zod DTOs and MUST match the endpoint set/behaviors in `openapi_specification.md`.
- R2. Implement the **core movement endpoints** exactly: `POST /movements` (delivery/refill-in), `PATCH /movements/{id}/return` (RENTAL → return-to-stock + `rental_days`; REFILL → close fill cycle without stocking — `014`), `PATCH /movements/{id}/swap`, `POST /movements/{id}/void`.
- R10. Implement **refill rate** endpoints (`014` / D-19): `GET/POST /refill-rates`, `PATCH /refill-rates/{id}` (`rates:read` / `rates:write`).
- R11. Implement `GET /reports/refill` for refill count/revenue aggregates (`007` / `014`).
- R3. Enforce **request validation** to `422 VALIDATION_FAILED` with field-level `details[]`; map business-rule violations to the specific `409`/`422` codes in `openapi_specification.md` §6.
- R4. Support **cursor pagination** (`limit`≤200, `cursor`), the **filter** grammar (`filter[field][op]`), and **sort** whitelists per collection.
- R5. Support **idempotency** (`Idempotency-Key` on creating POSTs) and **optimistic concurrency** (`ETag` + `If-Match` → `409 VERSION_CONFLICT`).
- R6. Emit the standard error envelope `{ error: { code, message, details[], request_id } }` for all failures.
- R7. Wrap each state-changing operation in a DB transaction and set the audit session GUCs (see `003`/`005`).

## Constraints

- C1. **Code-first with parity:** implement in NestJS with Zod DTOs; `@nestjs/swagger` emits the runtime OpenAPI JSON; the frontend `packages/api-client` is generated from that emitted JSON. A parity test asserts the emitted spec matches the design contract (`openapi_specification.md`) — endpoints, params, status codes, error codes. If they diverge, fix the code, not the test.
- C2. All timestamps ISO-8601; dates `yyyy-mm-dd` on the wire (UI localizes to `dd/mm/yyyy`). Zod schemas encode these formats.
- C3. Territory-scoped roles receive pre-filtered results; the server enforces scope regardless of client filters (Nest guard + repository predicate — see `005`).
- C4. No endpoint may bypass the business rules in `001`; the DB remains the last line of defense.
- C5. Reports/billing that are expensive run **async** (job handle + polling), per `007`.
- C6. Zod is the **only** validation mechanism; business-rule violations surfaced by the DB (constraint/trigger errors) are translated by a Nest **exception filter** into the mapped `409`/`422` codes — controllers never hand-validate.

## Acceptance Criteria

- AC1. Contract/parity tests validate every request/response against the Zod schemas and assert the Swagger-emitted OpenAPI matches `openapi_specification.md`.
- AC7. `GET /api/docs` serves Swagger UI and `/api/docs-json` returns a valid OpenAPI 3.1 document generated from the Zod DTOs.
- AC8. An invalid request body fails the global `ZodValidationPipe` with `422 VALIDATION_FAILED` and per-field `details[]` derived from the Zod issues.
- AC2. `POST /movements` on an already-out cylinder returns `409 CYLINDER_ALREADY_OUT` (BR-01); a `REFILL` on an `OURS` cylinder returns `422 KIND_BASIS_MISMATCH` (BR-08).
- AC3. `PATCH /movements/{id}/return` on a RENTAL returns the computed `rental_days` and rejects `return_date < delivery_date` with `422 RETURN_BEFORE_DELIVERY`; on a REFILL closes without stocking (`014` AC2).
- AC9. `POST /refill-rates` with an overlapping window for the same gas/capacity/unit returns `409 RATE_OVERLAP`.
- AC10. `GET /reports/refill` for a period returns refill_count and revenue consistent with billed fill lines when rates exist.
- AC4. Re-sending a POST with the same `Idempotency-Key` and identical body returns the original result, not a duplicate.
- AC5. A stale `If-Match` on any update returns `409 VERSION_CONFLICT`.
- AC6. List endpoints honor documented filters/sorts and reject unknown fields with `422`.

## Edge Cases

- Offline field sync: retried creates with the same idempotency key must be safe; a create that now conflicts (cylinder taken meanwhile) returns `409` for the client to resolve (see `006`).
- Void of an already-billed movement → `409 ALREADY_BILLED`.
- Swap where the returned serial is itself busy → `409 RETURNED_CYLINDER_BUSY`.
- Search query `< 2` chars → `422 QUERY_TOO_SHORT`.
- Large report exports → async job; endpoint returns `202` + handle.

## Dependencies

- `002` (domain), `003` (persistence + transactions + GUC), `005` (auth/authorization), `007` (report endpoints), `009` (rental/billing), `014` (refill rates + close semantics).

## Implementation Notes

- **NestJS layering:** `Controller` (routing + Swagger decorators + Zod DTOs) → `Service` (domain operations, transactions) → repository/data layer (SQL). Cross-cutting via Nest primitives: global `ZodValidationPipe`, an `HttpExceptionFilter` that renders the standard envelope, a `TransactionInterceptor` that opens the DB transaction and runs `SET LOCAL app.current_user_id/current_role_code/source` (audit GUCs, `003`/`005`), and a `PaginationInterceptor`/decorator for the cursor/filter/sort grammar.
- **Zod schemas** live in a shared module and are reused by DTOs (`createZodDto`), service-layer guards, and tests; keep request, response, and query-param schemas explicit per endpoint. The **filter/sort whitelists** are Zod enums so unknown fields fail validation (`422`).
- **Swagger:** call `patchNestjsSwagger()` (nestjs-zod) before building the document so Zod DTOs render; group by tag; document the standard error envelope and error codes as reusable responses.
- **DB access:** treat `schema.sql` as authoritative — use a SQL-first typed query builder (e.g. Kysely/Drizzle) or repositories; do **not** let an ORM own/sync the schema, constraints, triggers, or partitions.
- **Error mapping:** the exception filter maps Zod errors → `422`, DB unique/exclusion/check/trigger errors → the mapped `409`/`422` codes (e.g. exclusion `ex_move_no_overlap` → `CYLINDER_ALREADY_OUT`), keeping codes consistent with `openapi_specification.md` §6.
- Emit domain events (movement created/closed, cylinder sold/lost) via a Nest event bus for reporting read-models and notifications (`007`).
- Rate-limit (`@nestjs/throttler`) and require MFA on privileged endpoints (billing approve/export, admin) per `005`.
- **Settings:** `GET /settings` (capability `supplier_loans:read`) and `PATCH /settings` (`supplier_loans:write`, `If-Match` on aggregate `version`) expose `SystemSettings` from `@weld/schemas`: `supplier_loan_overdue_days`, `long_outstanding_days`, `business_timezone`, `rental_min_days`, `primary_language`. Partial PATCH allowed (at least one field). Persist in `system_setting` (003 R7; D-13 / D-14 / D-17).
